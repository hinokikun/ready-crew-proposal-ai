# Presentation Review Acceptance

Version 19.1 stabilizes the Presentation Review loop so that AI review results can be accepted, converted into a safe revision outline, approved by a human, and regenerated as a new Beautiful.ai presentation without overwriting the previous deck.

## Acceptance Scope

- Review results are structured as scores, short reasons, evidence summaries, confidence, human-review flags, and selectable actions.
- The UI requires users to select improvement actions before creating a revision.
- Beautiful.ai regeneration is never automatic. It is available only after a manager/admin approves the revision.
- Revision generation creates a new Beautiful.ai presentation ID and new editor/player URLs.
- Existing decks are not overwritten.
- Revision history is scoped by `organization_id` and `workspace_id`.

## Revision Flow

1. Proposal v1 is generated by the existing proposal workflow.
2. Presentation Review creates scores and safe improvement actions.
3. A member/admin selects actions and creates a draft revision.
4. A manager/admin approves or rejects the revision.
5. Approved revisions can be sent to Beautiful.ai as a new generated presentation.
6. The revision timeline stores v1, v2, and v3 metadata.
7. Re-review can be run after a generated revision; improvement is not treated as automatically successful.

## Status Model

Allowed revision statuses:

- `draft`
- `reviewing`
- `generation_requested`
- `generating`
- `generated`
- `approved`
- `rejected`
- `failed`
- `archived`

`PRESENTATION_MAX_REVISIONS` defaults to `3`.

## Stored Data

The backend stores:

- Review scores
- Short reasons and evidence summaries
- Confidence and human-review flags
- Selected actions
- Sanitized `PresentationOutline`
- Revision metadata
- Diff summaries
- Beautiful.ai presentation ID
- Editor/player URLs
- Created/approved/generated timestamps

The backend must not store:

- API keys
- Passwords
- Beautiful.ai tokens
- Full customer emails
- Full generated proposal text
- Raw Beautiful.ai response bodies

## Permission Matrix

- `member`: create review, select actions, create draft revision.
- `manager`: approve, reject, generate Beautiful.ai revision.
- `admin`: all actions.
- `viewer`: read only.

Frontend controls are convenience only. Backend routes enforce permissions.

## Acceptance Checks

- v1 and v2 have separate `beautiful_ai_presentation_id` values.
- Beautiful.ai mock mode returns a version-specific mock ID.
- Revision diff displays before/after summaries and human action.
- Workspace A cannot read Workspace B review/revision records.
- Learning receives aggregate metrics only.
- Presentation Analytics includes review count, revision count, success rate, approval rate, unresolved issues, and generation failures.

## Cloud Verification

Cloud checks are not considered complete until a human verifies:

- GitHub Actions success.
- Vercel build success.
- Render `/health` and `/health/ready`.
- `GET /api/beautiful-ai/status`.
- Beautiful.ai mock generation in production-like settings.
- Real Beautiful.ai generation only after account/API contract confirmation.
